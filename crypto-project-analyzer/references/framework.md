# 六步分析框架详细指南

## 第一步：项目概览

### 目标
快速了解项目全貌，建立基本认知。

### 采集内容
- 基本信息：名称、官网、Twitter、合约地址、所在链
- 项目定位：赛道、细分领域
- 核心机制：一句话描述项目怎么运作
- 技术架构：关键组件和数据流（用 ASCII 图）
- 关键角色：项目中的参与方及其职责
- 依赖关系：外部依赖及风险等级
- 市场数据快照：价格、FDV、市值、流动性、24h 交易量

### 数据源
1. 官网 → `raw/homepage-data.md`
2. DexScreener API → `raw/dexscreener-data.md`
3. 项目文档/白皮书
4. Twitter 主页

## 第二步：代币经济学

### 目标
通过智能合约和链上数据分析代币的分配、机制和安全性。

### 采集内容
- 代币参数：名称、符号、精度、总供应量、合约类型
- 发行平台：自部署 or 第三方（Doppler、Clanker、pump.fun 等）
- 代币分配：Top 持有者排名、占比、地址类型
- 预挖检查：vesting 参数、实际预挖金额
- 通胀机制：年通胀率、铸造权限、硬编码上限
- Owner 权限：可执行操作、是否已放弃所有权
- 流动性：池地址、是否锁定、锁定方式

### 数据采集流程

1. 运行 `collect_chain_data.sh` 获取基础链上数据
2. 读取合约源码（如已验证）分析关键参数
3. 用 DexScreener 浏览器抓取准确持有者数据
4. 交叉验证数据

### 关键注意事项
- Uniswap v4 使用单一 PoolManager，Blockscout holders_count 严重偏低
- 以 DexScreener 页面 Holders 数据为准
- 检查 Owner 是 EOA 还是合约
- 检查 Pool 地址是否为 dead address（流动性锁定）

### 输出
`02-代币经济学.md` + `raw/contract-source.md` + `raw/contract-data.md` + `raw/holders-and-market-data.md`

## 第三步：项目特性

### 目标
根据项目类型深入分析其核心机制。

### 按项目类型选择分析重点

| 项目类型 | 分析重点 |
|---------|---------|
| AI 挖矿 | 挖矿流程、挑战设计、积分系统、成本分析、收益估算 |
| DeFi | TVL、收益率、清算机制、智能合约风险 |
| GameFi | 游戏机制、代币消耗/产出、玩家经济循环 |
| NFT | 铸造机制、稀缺性、版税、二级市场 |
| 基础设施 | 技术架构、性能指标、采用率 |
| Meme | 叙事强度、社区热度、持仓集中度 |

### 数据源
- 官方文档/白皮书/skill.md
- API 端点实际测试
- 合约源码
- 实际体验（如果可行）

### 输出
`03-项目特性.md`（可加后缀如 `-挖矿机制`、`-DeFi机制`）

## 第四步：社区与生态

### 目标
评估社区基础、生态关系和团队背景。

### 采集内容

社交媒体（用浏览器抓取）：
- Twitter：粉丝数、帖子数、关注数、Bio、注册时间
- Discord：成员数、活跃度（如有）
- Telegram：成员数（如有）

链上活跃度：
- 持有者数量（DexScreener）
- 链上转账总数（Blockscout counters API）
- 24h 独立交易者数（DexScreener Makers）
- 买卖比

生态分析：
- 上下游关系图
- 集成方/合作方
- 发行平台背景

团队背景：
- 公开身份 or 匿名
- GitHub 仓库和活跃度
- 过往项目
- 融资信息

竞品对比：
- 同赛道 2-3 个项目横向比较

### 输出
`04-社区与生态.md`

## 第五步：风险综合评估

### 目标
汇总前四步所有风险，形成系统性评估。

### 风险矩阵模板

```markdown
| 风险类别 | 风险项 | 等级 | 影响 | 来源 |
|---------|--------|------|------|------|
| 中心化 | ... | 🔴/🟡/🟢 | ... | 0X-步骤名 |
```

### 风险类别
- 中心化风险：单点故障、闭源组件、托管依赖
- 合约风险：Owner 权限、未审计、自定义代码
- 市场风险：早期项目、低流动性、高波动
- 团队风险：匿名、无融资、无过往项目
- 机制风险：经济模型缺陷、死亡螺旋可能

### 场景分析
- 最佳情况
- 基准情况
- 最差情况
- 特殊风险场景（如死亡螺旋、大户垄断）

### 综合评级
加权评分：合约安全(20%) + 中心化(30%) + 市场(20%) + 团队(15%) + 机制(15%)

### 输出
`05-风险综合评估.md`

## 第六步：投资建议与参与策略

### 目标
给出可操作的建议。

### 内容
- 项目本质判断（一句话定性）
- 参与方式分析（列出所有可能的参与方式 + 各自的成本/收益/风险）
- 推荐策略（分阶段，含金额参考）
- 止损线
- 监控清单（指标 + 数据源 + 频率）
- 正面/负面信号
- 综合评分卡

### 评分维度（⭐1-5）

评分卡必须包含三列：维度、评分（星星）、说明（具体理由）。禁止只写星星不写说明。

| 维度 | 说明 | 评分示例 |
|------|------|---------|
| 创新性 | 机制是否新颖 | "Proof of Inference 概念新颖，用 LLM 推理替代传统算力" |
| 合约安全 | 代码质量和审计 | "标准模板 + OpenZeppelin + 流动性永久锁定" |
| 去中心化 | 核心机制去中心化程度 | "核心机制高度中心化（协调器+Bankr 双重链下依赖）" |
| 团队信任 | 透明度和背景 | "匿名团队，但有正规生态背书" |
| 社区基础 | 规模和活跃度 | "极早期（718 粉丝），缺乏沟通渠道" |
| 参与门槛 | 进入成本 | "低门槛（~$24 起步），无需硬件" |
| 短期潜力 | 近期上涨可能 | "AI 叙事热度 + 早期红利，但可能过热" |
| 长期潜力 | 可持续发展 | "取决于去中心化进程和持续运营" |

### 输出
`06-投资建议与参与策略.md`

## 报告信息完整性规则（硬性要求）

1. **合约地址**：必须显示完整 42 位地址，禁止截断（如 `0xA601...0BA3`）
2. **URL/链接**：所有网页地址必须完整显示，包括官网、API、DexScreener、Blockscout 等
3. **持有者地址**：Top 10 持有者地址必须完整展示
4. **关键合约汇总表**：报告末尾必须包含「关键合约与链接」表，汇总所有合约地址和相关 URL
5. **数据来源标注**：每个数据点标注来源（API/页面）和采集时间戳
6. **目的**：方便读者直接复制地址/URL 进行二次验证和查询

## 合并报告

完成六步后，生成两个版本的报告：

### 精简版 (`report-summary.md`)
- 每步提取核心发现、关键表格和结论
- 去掉详细分析过程，保留结果
- 适合快速浏览和分享
- 目标：200-300 行

### 完整版 (`report-complete.md`)
- 按顺序拼接 01-06 全部内容，用 `---` 分隔
- 不做任何精简或省略
- 适合深度阅读和二次验证
- 目标：保留原始报告全部内容

### 共同规则
- 所有地址和 URL 完整展示（不截断）
- 末尾包含「关键合约与链接」汇总表
- 综合评分卡必须包含三列（维度+星星+说明）
- 添加 YAML frontmatter（title, author, description）
- 添加免责声明

### 默认行为
- 用户说"输出报告" → 同时生成两个版本 PDF，都发给用户
- 用户说"精简版" → 只发精简版
- 用户说"完整版" → 只发完整版
